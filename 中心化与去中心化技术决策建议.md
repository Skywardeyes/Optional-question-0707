# 选做 B：CA/PKI 与 DID/VC 技术决策建议

## 术语说明

- CA（Certificate Authority，证书颁发机构）
- PKI（Public Key Infrastructure，公钥基础设施）
- DID（Decentralized Identifier，去中心化标识）
- VC（Verifiable Credential，可验证凭证）
- KYC（Know Your Customer，了解你的客户）
- AML（Anti-Money Laundering，反洗钱）
- DAO（Decentralized Autonomous Organization，去中心化自治组织）
- eIDAS（electronic IDentification, Authentication and trust Services，电子身份识别、认证与信任服务）
- GDPR（General Data Protection Regulation，通用数据保护条例）

## 情况一：考虑中国市场和监管环境

中国未来 5 到 10 年的身份基础设施仍会以 CA/PKI 为核心，特别是在政府、金融、电信、能源、工业互联网等强监管领域。DID/VC 更适合作为面向数据流通、隐私保护和跨组织协作的新型身份增强技术，而不是替代 PKI。

| 场景                         | 建议                             | 关键依据（涵盖信任模型、责任边界、撤销恢复、审计合规、接入成本） | 主要风险                                                     |
| :--------------------------- | :------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| 政府政务系统身份认证         | 优先采用 CA/PKI                  | **信任模型：** 中心化分级信任（国家根CA→政务CA），强监管且不可篡改。<br>**责任边界：** 国家/省级RA（注册机构）与CA承担明确的法律主体责任，便于追责。<br>**撤销恢复：** 依托CRL/OCSP（证书撤销列表/在线证书状态协议）实现实时吊销，流程法律效力明确。<br>**审计合规：** 全生命周期日志留痕，完全满足等保2.0、密码法及电子政务审计要求。<br>**接入成本：** 硬件（加密机）、软件及运维投入高，但现有政务系统改造平滑，生态成熟。 | CA体系建设和运维成本极高；证书生命周期管理复杂（如Ukey丢失）；跨部门互认需统一根节点。 |
| 金融行业身份认证             | 优先采用 CA/PKI，DID/VC 作为辅助 | **信任模型：** 以央行/金融CA为中心，满足银保监会强实名制监管架构。<br>**责任边界：** 金融机构对核身环节负责，CA对签名证书真实性负责，责任链清晰。<br>**撤销恢复：** 支持实时黑名单查询，冻结挂失流程与柜台核身绑定，安全等级高。<br>**审计合规：** 满足反洗钱（AML）、支付清算审计要求，交易留痕至少5年以上。<br>**接入成本：** 接口改造复杂，需兼容多品牌U盾；DID/VC辅助可降低KYC材料重复提交成本。 | CA私钥泄露可能导致系统性风险；传统U盾用户体验较差；VC法律地位在金融纠纷中尚存争议。 |
| 电信运营商网络身份认证       | 优先采用 CA/PKI                  | **信任模型：** 基于设备证书和SIM卡鉴权的中心化网络信任体系（GBA框架）。<br>**责任边界：** 运营商对管道内设备身份和接入行为承担合规责任。<br>**撤销恢复：** 支持空中写卡（OTA）远程吊销与重发，但依赖信令通道可靠性。<br>**审计合规：** 满足通信行业安全防护要求，信令溯源链条完整。<br>**接入成本：** 海量物联网卡证书管理需建设统一平台，老旧基站改造成本高。 | 海量设备证书管理复杂；证书过期批量更新可能引发网络抖动；密钥需硬件级保护。 |
| 政务数据共享与跨部门协同     | CA/PKI 为基础，DID/VC 逐步探索   | **信任模型：** 部门间采用前置系统互信（PKI互通），DID用于个人/企业数据标签映射。<br>**责任边界：** 数据提供方对源数据真实性负责，PKI保障传输通道抗抵赖。<br>**撤销恢复：** 人员证书随岗位变动实时吊销；VC可支持凭证粒度级撤销。<br>**审计合规：** PKI满足《数据安全法》审计要求，VC补充最小化披露合规项。<br>**接入成本：** 双体系融合增加技术复杂度（需维护传统CA和区块链/分布式账本两层）。 | 双体系融合增加技术债务；DID标准、凭证模板和治理体系尚不成熟，可能造成数据孤岛。 |
| 企业内部身份认证             | 优先采用 CA/PKI                  | **信任模型：** 企业内部根CA，适合明确的树状组织架构和行政区划。<br>**责任边界：** HR（人力资源）负责入离职身份源，IT负责证书签发，责任边界分明。<br>**撤销恢复：** 与AD/LDAP（活动目录/轻量级目录访问协议）联动，离职即时吊销，恢复需重新审批。<br>**审计合规：** 支持零信任架构的持续审计，记录员工访问敏感系统全流程。<br>**接入成本：** 与OA、ERP、VPN集成方案成熟，但证书续期操作影响员工效率。 | 证书申请、更新、吊销流程繁琐影响办公效率；设备遗失与HR离职不同步可能导致权限残留。 |
| 企业间供应链协同             | PKI 为基础，DID/VC 增强          | **信任模型：** 依托核心企业CA签发上下游接入证书（星型信任），DID补充多方互认凭证。<br>**责任边界：** 核心企业对接入节点负责，VC发行方对凭证内容（如质检报告）承担法律责任。<br>**撤销恢复：** 供应商退出时CA证书即时阻断API调用；VC可设置有效期短时凭证降低撤销压力。<br>**审计合规：** PKI保障数据防篡改，VC支持“仅出示合格/不合格”避免泄露商业机密。<br>**接入成本：** 中小供应商部署证书软硬件成本高；DID可降低合规证明流转的行政开销。 | 核心企业规则变化影响上下游；VC凭证真实性与线下治理强绑定，易伪造源头数据。 |
| 工业互联网设备身份认证       | 优先采用 CA/PKI                  | **信任模型：** 基于V2X/工控专用CA的集中式身份签发，适合高可靠生产网。<br>**责任边界：** 设备制造商预置证书，运维单位负责部署，主体责任通过证书链追溯。<br>**撤销恢复：** 支持按批次、按时间策略自动轮转证书；老旧设备需手动灌装密钥。<br>**审计合规：** 满足等保及电力/能源行业护网行动对通信加密和身份鉴权的强制审计。<br>**接入成本：** 存量老旧PLC（可编程逻辑控制器）改造需外挂加密模块，成本极高且影响生产连续性。 | 海量异构设备证书管理困难；设备算力不足以支撑复杂密码运算；证书轮转期间可能造成业务中断。 |
| 物联网开放生态               | 探索 DID/VC + PKI 融合           | **信任模型：** 采用“设备DID+云端PKI网关”的混合模式，终端自主生成密钥对，云端做身份绑定。<br>**责任边界：** 设备厂商对固件身份负责，平台方对接入鉴权负责，链路层依赖PKI加密。<br>**撤销恢复：** 依赖智能合约或分布式撤销列表，但低成本设备难以承载复杂状态同步。<br>**审计合规：** 隐私计算结合VC可实现设备属性（如位置、温度）的合规披露。<br>**接入成本：** 低端传感器仅需存储几个DID文档，大幅降低证书管理成本，但网关侧需适配。 | 标准（W3C/DIF）尚未统一；设备私钥被盗难以物理隔离；轻量级密码算法在MCU（微控制器）上性能受限。 |
| 电子合同、电子签章、电子票据 | 优先采用 CA/PKI                  | **信任模型：** 严格遵循《电子签名法》，依赖依法设立的第三方CA机构为信任锚点。<br>**责任边界：** 用户过错（如私钥泄露）用户担责，CA过错（如签发冒名证书）CA担责，司法鉴定清晰。<br>**撤销恢复：** 合同签署后不可撤销（除非司法判决）；证书吊销不影响已签署文件的法律效力。<br>**审计合规：** 完整的数字时间戳和操作日志，满足《档案法》电子归档要求。<br>**接入成本：** 需购买企业证书（OV/EV）和个人证书，集成PDF/OFD签名组件较成熟。 | 依赖CA机构服务可用性；证书使用流程（插Ukey/输密码）影响高频签署体验；跨CA互认存在壁垒。 |
| 学历、职业资格、技能证明     | 优先考虑 DID/VC                  | **信任模型：** 以教育/人社部门为权威发行方（Issuer），个人持有（Holder），企业按需验证（Verifier）的三角模型。<br>**责任边界：** 发行方对证书内容真实性承担行政/法律责任；持有方对私钥保管负责；验证方自行评估采信度。<br>**撤销恢复：** 支持按凭证ID定点撤销（如学位收回），保留历史变更日志，但尚无国家级统一撤销广播机制。<br>**审计合规：** 支持零知识证明验证“是/否毕业于985”，避免泄露身份证号等敏感信息。<br>**接入成本：** 发行方接入成本较低（部署Issuer节点），企业验证方可使用公开API或扫码枪，无需预置根证书。 | 权威发行方治理规则（如定级、换发）难统一；私钥丢失后的恢复机制法律界定模糊；验证方采信习惯培养周期长。 |
| 数字身份钱包与个人数据授权   | DID/VC 探索应用                  | **信任模型：** 用户自主掌控公私钥（Self-Sovereign），通过去中心化标识符映射多平台数据。<br>**责任边界：** 用户对授权行为负全责；服务方仅对数据真实性背书，不托管密钥。<br>**撤销恢复：** 依赖社交恢复或多签机制，技术门槛高；若种子丢失则身份永久丢失（无中心客服重置）。<br>**审计合规：** 满足《个人信息保护法》最小必要原则，授权记录可上链存证但需注意链上数据隐私。<br>**接入成本：** 需开发原生App/小程序钱包，普通智能手机即可运行，但需适配国产化操作系统（鸿蒙/欧拉）。 | 普通用户无法理解助记词和私钥概念；钱包丢失无客服找回；责任划分（被盗损失由谁承担）尚无司法判例。 |
| Web3、数字资产、虚拟空间身份 | DID/VC 优先，谨慎落地            | **信任模型：** 去中心化信任，依赖区块链底层（如长安链、星火链网）锚定DID文档。<br>**责任边界：** 智能合约代码承担部分逻辑责任，链上地址与现实身份解耦，合规实名难度大。<br>**撤销恢复：** 密钥轮换需更新链上DID文档，操作不可逆；不存在传统意义上的“证书吊销”。<br>**审计合规：** 需额外引入“实名DID”映射合规节点，否则无法满足国内网信办实名制监管要求。<br>**接入成本：** 开源框架（如Veramo、Indy）降低开发门槛，但需自建节点或依赖BaaS（区块链即服务）平台。 | 国内监管限制严格（虚拟货币禁令）；女巫攻击和地址买卖影响身份可信度；链上数据公开与隐私保护矛盾突出。 |

## 情况二：国际视野

本表与中国市场版本做差量对比：五个板块都必须保留；某一板块无明显差异时标注“与中国版本一致”；若五个板块都一致，则不单独列行。

| 场景                          | 建议                               | 关键依据（涵盖信任模型、责任边界、撤销恢复、审计合规、接入成本） | 主要风险                                                     |
| :---------------------------- | :--------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| 欧盟电子身份与信任服务        | eIDAS/PKI 为基础，DID/VC 逐步融合  | **信任模型：** 差异在于跨成员国互认，信任锚不只是单一国家或行业CA，而是eIDAS下的合格信任服务体系和欧洲数字身份钱包框架。<br>**责任边界：** 合格信任服务商、成员国监管机构、钱包提供方和凭证发行方之间需要按欧盟规则划分责任。<br>**撤销恢复：** 与中国版本一致，仍需要明确证书/凭证撤销状态、有效期和恢复流程。<br>**审计合规：** 差异在于除身份真实和签名有效外，还必须同时满足GDPR下的数据最小化、目的限制和用户权利要求。<br>**接入成本：** 差异主要来自跨成员国互操作、钱包适配和合规评估成本，而不是单纯部署CA。 | 欧盟各国实施节奏不统一；钱包互操作性不足；跨境凭证的法律效力认证流程冗长。 |
| 跨境金融、支付和合规身份      | CA/PKI + KYC/AML 凭证并用          | **信任模型：** 差异在于交易链路跨多个司法辖区，机构身份仍依赖成熟金融网络和PKI，KYC/AML VC只适合作为客户尽调材料的可验证载体。<br>**责任边界：** 汇款行、收款行、支付机构和虚拟资产服务商仍承担各自合规责任，不能因采信VC而转移KYC/AML最终责任。<br>**撤销恢复：** 与中国版本一致，仍需短有效期、黑名单/制裁名单联动和异常状态快速失效机制。<br>**审计合规：** 差异在于需要同时满足FATF旅行规则、制裁名单筛查、数据跨境传输和本地监管报送要求。<br>**接入成本：** 差异主要来自ISO 20022等报文体系、制裁筛查系统和多法域合规模型的改造。 | 不同司法辖区对披露字段、保存期限和监管报送要求可能冲突；制裁名单实时更新与静态凭证存在时间差；VC不改变金融机构根本法律责任。 |
| 全球供应链、碳足迹和产品溯源  | PKI 保障通信，DID/VC 表达凭证      | **信任模型：** 差异在于信任链跨越多个国家、认证机构、物流节点和行业联盟，VC更适合表达产品认证、碳数据和审计材料。<br>**责任边界：** 差异在于数据采集方、检测机构、物流方、品牌方和进口监管方责任更分散，需在凭证模板中明确谁对哪类数据负责。<br>**撤销恢复：** 与中国版本一致，仍需设备证书轮转、凭证有效期和异常批次召回机制。<br>**审计合规：** 差异在于需要兼容欧盟CBAM等跨境审计格式，以及不同国家对碳足迹、产地和供应链尽调的要求。<br>**接入成本：** 差异主要来自多级供应商的技术能力差异、跨境数据合规和凭证模板统一成本。 | 数据源真实性仍依赖线下物理防篡改；各国碳因子库口径不一；中小供应商难以承担复杂接入和合规成本。 |
| 国际教育、职业资格和移民材料  | 优先 DID/VC                        | **信任模型：** 差异在于验证方通常不在发行方所在国家，DID/VC可减少传统公证、邮件核验和人工背调。<br>**责任边界：** 差异在于学校、考试机构、职业协会、移民部门和用人单位各自采信规则不同，验证通过不等于自动承认资格。<br>**撤销恢复：** 差异在于跨国撤销与更新更依赖公开状态列表、短有效期凭证或发行方在线状态服务，而不是单一国内平台。<br>**审计合规：** 差异在于需要兼顾GDPR、FERPA等教育和隐私规则，尽量只披露学位、资格、有效期等必要字段。<br>**接入成本：** 与中国版本一致，DID/VC仍有助于降低重复证明、人工核验和材料流转成本。 | 发行方权威性识别困难；撤销和更新可能受时区、语言、制度差异影响；翻译、学历认证框架仍需要额外桥接层。 |
| Web3 链上身份、DAO 和空投治理 | DID/VC 优先，配合链上风控          | **信任模型：** 差异在于国际Web3生态更依赖公链、钱包地址、智能合约和社区声誉，而不是实名机构信任根。<br>**责任边界：** 差异在于智能合约不能替代法律责任，项目方仍需对凭证发行规则、空投规则和用户风险提示负责。<br>**撤销恢复：** 差异在于链上凭证撤销通常受交易成本、密钥轮换和钱包恢复机制影响，不能照搬传统证书吊销模型。<br>**审计合规：** 差异在于需要额外接入链上分析、制裁地址筛查和反女巫攻击机制。<br>**接入成本：** 与中国版本一致，开源工具降低开发门槛，但链上交易费、节点服务和安全审计仍有成本。 | 钱包地址不等于真实身份；女巫攻击、地址买卖、钓鱼授权和隐私泄露风险高；不同国家对虚拟资产监管差异极大。 |
