## 结论先行

**建议在“跨组织凭证互认”场景中采用 CA/PKI 与 DID/VC 分层并存方案：CA/PKI 用于机构接入、系统身份和通信安全，DID/VC 用于业务凭证的签发、持有、出示和验证。**

本建议不预设 CA/PKI 与 DID/VC 的主次关系，而是按问题分工：跨组织系统之间既要解决“谁在接入、通道是否可信、操作能否审计”，也要解决“凭证由谁签发、内容是否被篡改、是否仍然有效”。前者更适合由 CA/PKI 承担，后者更适合由 DID/VC 承担。

## 场景定义

选择场景：**跨组织凭证互认**。

典型例子：核心企业、供应商、检测机构、物流机构之间需要互认资质证明、质检报告、授权文件或合规证明。各方不一定属于同一个组织，也不一定共享同一套账号体系，但需要判断：

1. 对方机构或系统是否可信。
2. 凭证是谁签发的。
3. 凭证内容有没有被篡改。
4. 凭证是否仍然有效。
5. 发生争议时能否追责和审计。

### 典型业务例子

跨组织凭证互认可以发生在政府、金融、企业生态等多类场景。共同点是：一个组织签发某种证明，另一个组织需要在不重复人工核验的情况下判断它是否可信、是否有效、是否可追责。

| 例子                                                         | 凭证含义                                                     | 更接近的技术表达                                             |
| ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 市场监管、税务、人社等部门向企业签发电子证照或经营资质证明，其他政务部门办事时需要核验 | 政府部门声明某企业具备某项登记状态、经营许可、纳税信用或社保缴纳状态 | 业务凭证，适合表达为 VC；政务系统接入、电子签章和传输安全可使用 CA/PKI |
| 人社或教育部门签发职业资格、学历、培训合格证明，用人单位或其他部门按需核验 | 权威部门声明某个人具备某项资格、学历、培训记录或岗位准入条件 | 业务凭证，适合表达为 VC；签发方系统身份和查询接口可使用 CA/PKI |
| 银行、支付机构或征信机构之间复用 KYC、企业尽调、授信材料或合规证明 | 金融机构声明某客户已完成某类核身、尽调、风险评级或材料校验   | 业务凭证，适合表达为 VC；金融专网接入、机构证书、交易签名仍可使用 CA/PKI |
| 供应链金融中，核心企业、仓储方、物流方、银行之间互认订单、仓单、物流状态或应收账款证明 | 参与方声明某笔贸易、库存、运输或应收账款事实存在且处于某种状态 | 业务凭证，适合表达为 VC；接口调用、机构身份和关键交易签名可使用 CA/PKI |
| **A 公司产品通过 B 公司软/硬件兼容性测试，B 公司签发互认证证书** | **B 公司声明 A 产品在指定版本、指定环境下可流畅运行，并达到某项兼容或性能要求** | **业务凭证，适合表达为 VC；签发平台、验证 API、电子签章可使用 CA/PKI** |
| **公司某产品取得官方检测中心认证**                           | **检测中心声明该产品通过某项检测标准或合规测试**             | **业务凭证，适合表达为 VC；如果出具电子签章 PDF，可使用 CA/PKI 做签名** |
| **公司获得某合作伙伴或行业组织的资质认证**                   | **发行方声明该公司具备某项合作资质、服务能力或生态伙伴等级** | **业务凭证，适合表达为 VC；机构接入和系统通信仍可使用 CA/PKI** |

这里容易混淆的一点是：**权威机构签发的业务认证不等于 CA/PKI 里的数字证书。** 如果签发内容是“某个公钥属于某个主体”，才是典型 CA/PKI 证书；如果签发内容是“某个产品、公司或人员具备某项资质、能力、结果或授权”，则更接近 DID/VC 讨论的可验证凭证。

## 对比维度

| 维度     | 只用 CA/PKI                                                  | 只用 DID/VC                                                  | 分层并存方案                                                 |
| -------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 信任模型 | 以 CA 为信任锚，适合证明机构、服务器、用户或设备身份。跨组织互信需要统一根 CA 或交叉认证。 | 以发行方、持有方、验证方为核心，适合表达“谁签发了什么凭证给谁”。但验证方仍要判断发行方是否权威。 | CA/PKI 处理接入信任和通道信任；DID/VC 处理凭证信任和属性证明。 |
| 责任边界 | CA 对证书签发和身份绑定负责，机构对业务数据负责。证书能说明“谁接入”，但不能天然说明“凭证内容由谁负责”。 | VC 发行方对凭证内容负责，持有方负责保管和出示，验证方负责采信判断。但线下数据真实性仍依赖治理规则。 | 机构身份由 CA/PKI 固定，凭证内容责任写入 VC 模板和发行规则，争议时可区分“接入责任”和“内容责任”。 |
| 撤销恢复 | CRL/OCSP 等机制成熟，适合吊销机构证书、用户证书和 API 接入证书。 | 可按凭证 ID、状态列表或短有效期做撤销，但钱包丢失、密钥恢复和撤销广播机制仍需设计。 | 机构退出或系统失信时吊销证书；单个资质、报告、授权失效时撤销 VC，避免为业务凭证频繁重签证书。 |
| 审计合规 | 证书生命周期、签名、访问日志和系统操作容易纳入现有审计体系。 | 支持最小化披露和可验证流转，但链上/链下状态、隐私保护和采信规则需要额外治理。 | 访问链路、API 调用和机构身份走 PKI 审计；凭证签发、出示、验证和撤销走 VC 审计。 |
| 接入成本 | 体系成熟，但跨组织统一根、证书发放、硬件介质和接口改造成本较高。 | 凭证流转灵活，验证方接入轻，但 DID 方法、钱包、凭证模板和治理规则需要新建。 | 先复用现有 PKI 接入能力，再在高价值凭证上试点 VC，降低一次性改造成本。 |

## 技术建议

1. **机构和系统接入层使用 CA/PKI。**
    核心企业、供应商、检测机构等组织先通过证书完成机构身份认证，API 调用、mTLS、服务端身份校验和传输加密继续使用成熟 PKI 能力。
2. **业务凭证层使用 DID/VC。**
    质检报告、产品互认证书、官方检测认证、授权证明、供应商资质、合作伙伴资质、合规证明等材料由发行方签发为 VC。持有方按需出示，验证方校验发行方、签名、有效期、撤销状态和凭证字段。
3. **关键凭证优先短有效期。**
    对高风险或变化频繁的凭证，不完全依赖事后撤销，优先使用短有效期加在线状态查询，降低状态同步失败带来的风险。
4. **先试点，再扩展。**
    首批只选择一种凭证类型，例如“供应商资质证明”或“质检合格证明”。跑通模板、签发、验证、撤销、审计后，再扩大到更多凭证。

## 主要风险

| 风险               | 影响                                                      | 应对                                                   |
| ------------------ | --------------------------------------------------------- | ------------------------------------------------------ |
| 发行方权威性不清   | 验证方不知道是否该信某个 VC 发行方。                      | 建立发行方白名单、准入规则和凭证模板注册机制。         |
| 源头数据可能造假   | VC 只能证明凭证未被篡改，不能保证线下检测或录入一定真实。 | 将线下审核、检测流程、人员责任和抽检机制纳入治理。     |
| 凭证撤销状态不同步 | 已失效凭证可能继续被验证通过。                            | 使用短有效期、在线状态查询和定期同步撤销列表。         |
| 钱包或私钥丢失     | 持有方无法出示凭证，或凭证被冒用。                        | 设计企业级托管、密钥恢复、多签或重新签发流程。         |
| 双体系增加复杂度   | PKI、DID、VC、API 权限和审计日志并存，排障难度上升。      | 明确分层边界：PKI 管接入，VC 管凭证；统一日志关联 ID。 |

## 限制与前提

这个方案解决不了所有问题：

- **不能保证源头数据天然真实。** DID/VC 只能保证凭证签名可验证、内容未被篡改；检测、审核、录入环节仍要靠线下治理和责任约束。
- **不能替代企业接入认证。** VC 适合表达业务证明，不应替代 mTLS、API 鉴权、权限控制和网络安全策略。
- **不能自动解决法律采信。** 凭证是否被司法、监管或合作方认可，还需要合同、制度、行业规则和审计流程配套。
- **不能回避密钥管理。** 不论 CA/PKI 还是 DID/VC，私钥保护、轮换、丢失恢复和人员离职处理都必须设计。

落地前提：

1. 明确参与方角色：发行方、持有方、验证方、平台运营方、审计方。
2. 明确凭证模板：字段、有效期、签发条件、撤销条件和责任声明。
3. 建立发行方准入与信任目录。
4. 建立证书、凭证、API 调用和业务单据之间的统一审计关联。
5. 先在单一凭证类型上试点，验证后再推广。

## 最终建议

跨组织凭证互认不宜简单选择“CA 取代 DID”或“DID 取代 CA”。更客观的团队技术建议是：**按层分工、并存使用。CA/PKI 解决组织和系统接入关系，DID/VC 解决跨组织业务凭证表达与验证。** 这种方案既保留现有合规和审计优势，也能降低凭证重复提交、人工核验和过度披露的问题。

## 多角色评审

### 企业 CTO 视角：这方案是否可落地？

总体可落地，但应按“小范围试点、分层接入、逐步扩展”的方式推进，而不是一次性替换现有系统。

从 CTO 角度看，方案的可落地点在于：CA/PKI 不需要推倒重来，仍然用于机构身份、系统接入、mTLS、API 通信和电子签章；DID/VC 先选一个业务凭证类型试点，例如供应商资质证明、产品互认证书或质检合格证明。这样既能复用已有基础设施，也能验证 DID/VC 在凭证签发、出示、验证和撤销上的价值。

落地难点主要有四个：

1. 需要建设或采购 VC 签发、持有、验证和状态查询能力。
2. 需要统一凭证模板，否则验证方无法稳定解析字段。
3. 需要把 VC 验证结果接入现有业务系统，而不是停留在“能验签”的演示层。
4. 需要建立运维流程，包括密钥托管、凭证撤销、日志关联、异常处理和版本升级。

因此，CTO 视角下的建议是：**先做单一凭证、单一业务链路、少数合作方的试点，再判断是否扩大。**

### 安全负责人视角：风险有没有说透？

目前方案已经覆盖了主要风险，但安全负责人还会继续追问“签名有效之后怎么办”。因为 VC 的密码学验证只能证明凭证未被篡改、确实由某个私钥签发，不能自动证明源头事实真实。

还需要重点补充和持续关注：

1. **发行方准入风险**：谁可以签发哪类凭证，必须有白名单、准入审批和退出机制。
2. **密钥风险**：发行方私钥、机构证书私钥、钱包私钥一旦泄露，可能导致伪造凭证或冒用凭证。
3. **撤销不同步风险**：凭证已失效但验证方没有及时查到撤销状态，会造成错误采信。
4. **模板滥用风险**：同样是“资质证明”，字段边界、有效范围、测试版本、适用场景必须写清楚。
5. **日志割裂风险**：PKI 接入日志、VC 签发日志、VC 验证日志和业务操作日志如果无法关联，事后追责会困难。

安全负责人视角下的结论是：**风险不是只靠密码学解决，而要靠密钥管理、发行方治理、撤销机制、日志关联和应急流程一起解决。**

### 法务/合规视角：责任边界是否清楚？

责任边界基本清楚，但需要在制度和合同里写成可执行条款。技术上区分“接入责任”和“内容责任”是合理的：

- CA/PKI 负责证明机构、系统、用户或设备身份，支撑通信安全、电子签名和审计留痕。
- DID/VC 负责表达某个发行方对某个主体的业务声明，例如资质、检测结果、授权、KYC 结果。

但法务/合规会继续追问：

1. 凭证发行方对哪些字段真实性负责？
2. 持有方是否允许转交、代持或二次展示凭证？
3. 验证方采信凭证后，如果发生损失，责任如何分摊？
4. 凭证撤销后，已经基于凭证完成的业务动作是否仍然有效？
5. 凭证涉及个人信息或商业秘密时，是否满足最小必要和授权使用要求？

因此，法务/合规视角下的建议是：**技术方案只能提供可验证能力，最终采信规则、责任分摊、证据效力、隐私边界仍需通过合同、制度和审计规则确定。**

### 反方视角：为什么不只用 CA/PKI？

只用 CA/PKI 的问题是，它更擅长证明“某个公钥属于某个主体”，但不擅长表达复杂业务凭证。

例如产品互认证、官方检测认证、供应商资质、KYC 尽调结果，本质上是在表达“某主体具备某项属性、资格、结果或授权”。如果全部用 CA/PKI 承载，容易出现几个问题：

1. 证书用途被扩大，数字证书既要证明身份，又要承载业务属性，边界变混。
2. 业务属性变化频繁时，可能需要频繁重签或重发证书。
3. 凭证字段难以按场景选择性披露，容易过度暴露材料。
4. 跨组织之间如果没有统一根或交叉认证，接入和互认成本较高。

所以，不只用 CA/PKI 的理由是：**CA/PKI 适合做身份和通道的信任底座，但业务凭证流转需要更灵活的凭证表达层。**

### 反方视角：为什么不只用 DID/VC？

只用 DID/VC 的问题是，它不能替代所有企业接入和通信安全能力。

DID/VC 可以验证凭证是谁签发的、有没有被篡改、是否过期或撤销，但它不能完整替代：

1. mTLS、VPN、API 网关等系统接入控制。
2. 企业已有 CA、证书、电子签章和审计体系。
3. 强监管场景下已经成熟的证书生命周期管理。
4. 机构服务器身份、设备身份、服务到服务通信身份。
5. 线下审核、检测、尽调流程的真实性治理。

所以，不只用 DID/VC 的理由是：**DID/VC 适合表达和验证业务凭证，但机构接入、通信安全、强审计和既有合规体系仍需要 CA/PKI 或同类基础设施支撑。**

### 评审后结论

从四个视角综合看，分层并存方案仍然成立：

- CTO 视角关注能不能落地，结论是可以试点，但要小步推进。
- 安全负责人视角关注风险有没有闭环，结论是要补齐密钥、撤销、日志和应急。
- 法务/合规视角关注责任是否可执行，结论是要把技术边界写进制度和合同。
- 反方视角说明单用 CA/PKI 或单用 DID/VC 都会出现能力边界问题。

因此，更稳妥的方案不是替代关系，而是：**CA/PKI 保障接入、通信和审计底座；DID/VC 承载跨组织业务凭证表达和验证。**
