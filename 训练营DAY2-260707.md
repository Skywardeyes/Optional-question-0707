# 安全训练营 DAY2：加密、哈希、MAC、签名、证书、PKI 如何拼成一套系统

## 1. 从公司外访问内网：通道与身份

用户在公司外部，通过受保护通道 VPN 访问内网应用。应用侧还需要进行身份校验，例如：

- 用户名口令
- 证书认证
- OTP
- SSO

整体过程可以拆成两层：

1. **建立通道**：解决“怎么进去”。
2. **应用认证**：进了内网之后，还要再确认一次身份，判断“你是谁、有没有权访问”。

> [!important] 注意
> 连上 VPN，不等于应用已经登录。VPN 解决通道问题，应用认证解决访问者身份问题。

## 2. 两类需要确认的问题

| 类别 | 要确认的问题 |
| --- | --- |
| 通信 | VPN 入口是真的吗？链路会被偷看吗？内容会被改吗？公钥到底属于谁？算法以后还换得动吗？ |
| 访问 | 谁在访问？怎么证明是本人？能访问什么？这次访问正不正常？审计日志是否完整？ |

这两类问题不是割裂的，它们在同一条访问路径上前后相接。

## 3. 密码基础：不要把它当成算法清单

密码基础不是孤立的算法名，而是一串接连要回答的问题：

1. 内容能不能不让别人看懂？
2. 要加密，可双方没约定过密钥，密钥从哪来？
3. 密钥建好了，大量数据怎么加密才高效？
4. 内容加密了，会不会被悄悄改过？
5. 用的那把公钥，到底属于谁？

这些问题连起来，才会形成 TLS、VPN、HTTPS、mTLS 这些工程方案。

常见技术名词：

- AES
- SM4
- RSA
- ECC
- SM2
- DH
- ECDHE
- SM3
- SHA-256
- HMAC
- 数字签名
- X.509
- CA
- PKI

## 4. 加密、哈希、签名、证书分别解决什么

外部网络不可信，第一件事是内容不能让别人看懂。

| 机制 | 作用 |
| --- | --- |
| 加密 | 藏内容 |
| 哈希 | 只算摘要 |
| 签名 | 证明来源 |
| 证书 | 证明公钥归属 |

第一反应可能是 Hash、签名、证书，但它们都不藏内容。

- **Hash** 只算摘要，能发现“内容变了”，但原文照样能被读到。
- **签名** 证明“谁确认过这段话”。
- **证书** 证明“公钥属于谁”。
- **真正藏内容的是加密**。大量传输通常用对称加密，例如 AES、SM4。

GCM、CTR、CBC 这些分组模式今天只点名，不展开。

## 5. 密钥从哪里来

加密要用密钥，可客户端和 VPN 入口以前从没约定过密钥，密钥从哪来？

| 方案 | 问题 |
| --- | --- |
| 明文发送密钥 | 一发就泄漏 |
| 写死在配置里 | 一漏全漏，轮换困难 |
| 非对称协商 | 商量出会话密钥 |

扩展学习：

- PLS
- KEM：后量子密码兴起后的重要技术方向

非对称密码和密钥协商让两个素不相识的对象，有机会安全地商量出一把会话密钥。

- 公钥可以公开。
- 私钥必须保密。
- RSA、ECC、SM2、DH、ECDHE 会在这里出现。
- KEM 后面讲 PQC 还会再见。

## 6. 拿到一个公钥，凭什么就信

公钥可以公开，这是好事。但拿到一个公钥，能直接信吗？

中间人攻击中，攻击者也能生成密钥，并发来“自己的公钥”。所以关键不只是“有没有加密”，还要确认“和谁加密”。

- 如果没法确认公钥属于谁，就可能和假对象建立一条“看起来加密”的通道。
- 加密可能是真的，通信对象可能是错的。
- “公钥到底属于谁”这个问题，后面要靠证书来回答。

> [!note] 关键结论
> 算法强度解决不了对象冒充问题。公钥和身份必须能够绑定起来。

## 7. 混合密码体制：工程取舍

性能口径示例：

| 类型 | 示例性能 |
| --- | --- |
| 对称加密 | AES-128-GCM 约 292 万次每秒，16B 分组口径 |
| 非对称签名 | RSA-2048 签名约 367 次每秒，私钥签名口径 |
| 密钥协商 | P-256 ECDH 约 1798 次每秒，密钥协商口径 |

因此：

- 非对称能帮我们建立安全通信，但不适合拿来加密每一个数据包。
- 原因很实在：非对称安全，但计算成本高。
- 每段 JSON、每个页面资源都用非对称保护，性能和成本都扛不住。
- 常见做法是混合：非对称负责把会话密钥安全建起来。
- 后面大量数据交给对称加密保护，这就是混合密码体制。

## 8. Hash、MAC、签名

内容藏起来之后，还要回答两个问题：

1. 怎么知道它没被悄悄改过？
2. 怎么知道是谁发的？

三种方式的证明强度逐步增强，但适用场景不同。

| 方式 | 能证明什么 |
| --- | --- |
| Hash | 只管内容变没变 |
| MAC | 能证明是知道共享密钥的一方发过 |
| 数字签名 | 可向第三方证明 |

- **Hash** 能发现内容有没有变，但证明不了是谁发的。
- **MAC** 带密钥，能证明“知道共享密钥的一方发过”，但向第三方说不清具体是谁。
- **数字签名** 最强：私钥签、公钥验，前提是验签公钥可信。

这里不是比较谁更高级，而是看它们解决的问题和适用场景。

## 9. 证书：回答“公钥到底属于谁”

签名要验公钥，那这把公钥到底属于谁？证书就是来回答这个问题的。

证书核心内容：

```text
主体身份 + 公钥 + CA 签名
```

客户端需要分别检查：

1. 证书链能否追到可信根。
2. 证书是否没有过期。
3. 用途是否匹配。
4. 主机名是否匹配。
5. SAN 优先于 CN。

> [!warning] 容易踩坑
> 证书链验证通过，只说明信任链成立；主机名是否匹配，还要单独检查。

## 10. 企业里的三类证书

证书在企业里不止一个方向，不能混在一起。

| 证书类型 | 作用 |
| --- | --- |
| VPN 入口证书 | 确认 VPN 入口身份 |
| 内网应用证书 | 确认应用服务器身份 |
| 用户 / 设备证书 | 确认访问者身份 |

- **VPN 入口证书**：让客户端确认，我连到的确实是公司的 VPN 入口。
- **内网应用证书**：让浏览器确认，我访问的确实是这个应用服务器。
- **用户或设备证书**：反过来，让访问者证明自己是被授权的人或设备。

前两个用来确认服务端身份，第三个用来确认访问者身份。

证书不止是通信里的材料，它也能承载身份。因此，它把密码基础部分和身份安全部分连了起来。

## 11. 密码敏捷性

今天安全的算法，十年后还安全吗？

历史上很多以前安全的算法已经不安全了：

```text
DES -> 3DES -> MD5 -> SHA-1
```

算法会持续变化，算力、密码分析、攻击技术都会进步。所以写代码、设计协议、定数据格式时，不要把算法名写死在里面。

**密码敏捷性**：既然算法会老，真到了要换的那天，系统是否换得动？

| 维度 | 要点 |
| --- | --- |
| 盘点 | 知道哪里用了什么算法、密钥、证书 |
| 抽象 | 业务代码走抽象层，例如 OpenSSL EVP / PKCS#11 |
| 协商 | TLS、VPN、SSH 能协商算法套件 |
| 策略 | 算法选择、密钥长度、禁用清单靠配置管理 |
| 生命周期 | 证书和密钥的签发、轮换、吊销自动化 |
| 灰度 | 新旧算法能并存一段时间，能迁移、能回滚 |

例子：

```text
GMT 0018-2012:
SDF_GenerateKeyPair_RSA
SDF_GenerateKeyPair_ECC
```

如果算法进入函数名，迁移时调用层也要改。

## 12. PQC 与量子通信

- **PQC 抗量子密码**：经典计算机上运行的抗量子算法。
- **量子通信**：通信物理机制，不是算法族。

目前还没有可用的量产量子计算机，更多停留在威胁与预研阶段。

国际标准化方向：

- ML-KEM
- ML-DSA
- SLH-DSA
- Falcon
- HQC

国内方向：

- CTRU / CNTR
- AIGIS-ENC / AIGIS-SIG
- LMS-SM3 / HSS-SM3

国内目前还在算法征集阶段，标准还没定下来。公司很多产品已经有抗量子功能，但目前主要走软实现。

## 13. 工程协议里的组合

前面的机制，最终会落到大家天天用的工程协议里。

```text
证书与身份
-> 校验证书链 / 主机名 / 用途
-> 密钥协商
-> 对称加密保护数据
-> 完整性防篡改
```

TLS 能力组合：

```text
TLS -> HTTPS / mTLS
VPN -> TLS 或 IPsec 等形态
```

HTTPS 和 mTLS 常见地基是 TLS；VPN 可以使用 TLS，也可以使用 IPsec 等协议形态。

共同点：

1. 先确认对端身份。
2. 再建立会话密钥。
3. 客户端验证证书链、主机名和证书用途。
4. 双方商量出一把会话密钥。
5. 后面大量数据用对称加密保护。
6. 协议还要保证完整性、防篡改。

TLS 是一组密码能力的工程组合；VPN、HTTPS、mTLS 是不同层面的使用方式。

## 14. 访问管理

身份安全可以顺着一次访问来看：

| 环节 | 关注点 |
| --- | --- |
| 身份 | 公司里有没有这个人，是否在职 |
| 认证 | 验证证书是否有效，是否有对应私钥 |
| 授权 | 判断能不能进应用，能不能执行关键操作 |
| 应用接入 | 把应用入口和账号映射纳入统一管理 |
| 风险 | 判断设备、时间、地点是否正常 |
| 审计 | 留下登录、证书使用和关键操作痕迹 |

## 15. 身份技术的发展方向

目前身份技术还在往三个方向走：

1. **访问主体在扩展**：人、设备、服务、API、AI Agent。
2. **策略在动**：身份、设备、环境、行为、风险持续评估，例如短期证书或事件证书，这就是零信任口径。
3. **信任模型增加**：CA/PKI 与 DID、VC 分场景出现。

三个边界特别容易混：

- VPN 不是 SSO。
- SSO 也不是授权。
- 证书认证不是换了个名字的密码，它靠的是持有私钥。

## 16. CA / PKI 与 DID / VC

常见误区：DID 更先进，所以会取代 CA。这个判断不准确。

| 模型 | 特点 | 适用方向 |
| --- | --- | --- |
| CA / PKI | 中心化信任，责任边界清楚，合规成熟，责任明确 | 企业内网、金融、政务、电子签名 |
| DID / VC | 跨组织身份，可验证凭证，隐私保护 | 需要具体评估治理、恢复、生态和合规的场景 |

未来更可能是按场景分工、局部融合，而不是一方把另一方替掉。

## 17. 全链路串讲

不引入新概念，串起完整访问链路。

### 17.1 从公司外到内网

```text
公司外 -> 对称加密 -> 密钥协商 -> 会话密钥 -> 公司内网
```

### 17.2 证书连接通信可信与身份可信

```text
服务端证书 -> 证书链 -> 主机名 -> 用户/设备证书 -> 身份可信
```

### 17.3 认证之后继续判断

```text
认证 -> 授权 -> 风险 -> 二次认证 / 阻断 -> 告警
```

### 17.4 审计能否还原过程

```text
谁 -> 什么时候 -> 从哪里 -> 做了什么 -> 能否还原
```

## 18. 课后任务

### 18.1 必做题

给一家 200 人公司设计身份认证系统骨架。

重点：用一套设计骨架，把访问链路里的身份判断讲清楚。

需要完成：

1. **画身份流**：从用户 / 设备开始，画到 VPN、认证入口、应用、授权、风险、审计。标出每一步谁在验证什么。
2. **做接入选择表**：列几类应用，选择口令 + OTP、SSO、证书或 mTLS，并写清理由。
3. **风险和对策表**：至少覆盖密码泄露、证书丢失、离职账号、权限过大、日志缺失。

### 18.2 选做 A：TLS 证书链观察 + 最小客户端实验

目标：亲手看到证书链验证、主机名校验、证书用途校验分别在管什么。

步骤：

1. 任选一个 HTTPS 站点，或用本地自签证书起一个测试服务。建议先用真实站点跑通。
2. 用 `openssl s_client` 观察服务端证书、链路证书、SAN、有效期、用途和验证结果。
3. 用 Python / Go / Java / `openssl verify` 任选一种方式，写一个最小客户端或验证命令。
4. 改错主机名、换不受信根、去掉中间证书，任选一种，记录报错和原因。

交付：

- 命令或代码
- 关键输出
- 一次成功验证
- 一次失败验证
- 自己的结论

### 18.3 选做 B：CA 和 DID 的技术决策建议

请按团队技术建议来写：给出场景、选择、理由和风险。

可选场景：

- 企业内网应用证书登录
- 跨组织凭证互认
- 员工资质证明

至少比较：

- 信任模型
- 责任边界
- 撤销恢复
- 审计合规
- 接入成本

可以推荐 CA/PKI、DID/VC，也可以建议分场景并存，但必须写清理由。

还要说明：

- 这个选择解决不了什么。
- 后续落地还需要哪些前提。

交付：

- 一页决策建议，或
- 一张对比表

要求：结论先行，后面跟依据和风险。
